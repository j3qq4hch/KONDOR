# Описание возможностей нового EDA инструмента
*Рабочий документ. Версия для обсуждения с потенциальными участниками проекта.*

---

## Философия

Существующие EDA-инструменты делятся на два лагеря: бесплатные с архитектурными ограничениями (KiCad) и дорогие с раздутой кодовой базой и тормозным интерфейсом (Altium). Eagle занимал разумную середину, но уничтожен Autodesk. Облачные решения (EasyEDA) решают часть проблем ценой потери контроля над данными.

Новый инструмент строится на нескольких принципах, которые не подлежат компромиссу:

- **Скорость — не опция, а архитектурное свойство.** Рендеринг на GPU, инкрементальные алгоритмы, spatial indexing везде где нужен spatial lookup.
- **Локальный, без облака.** Данные проекта принадлежат инженеру.
- **Открытый формат файлов.** XML или JSON, самодокументированный, пригодный для хранения в git и обработки внешними скриптами.
- **Constraint-based геометрия.** Произвольные углы и геометрические связи — не костыль, а основа редактора.
- **3D — первоклассный гражданин**, а не внешний просмотрщик.

## Редактор схем

Стандартный функционал реализован полностью: иерархические схемы, шины с правильной семантикой, net labels, живая синхронизация со схемой платы.

**Принципиальные отличия:**

- ERC работает инкрементально, в реальном времени. Нарушение подсвечивается в момент создания, а не после нажатия кнопки.
- Варианты сборки (assembly variants) встроены в модель данных с первого дня, а не прикручены поверх. Один проект — несколько BOM для разных конфигураций продукта.
- Иерархические блоки с параметрами, пригодные для переиспользования между проектами.

## Редактор платы

### Constraint-based геометрия

Это главное архитектурное отличие от всех существующих инструментов в данном ценовом сегменте.

Вместо сетки как единственного способа задать положение объекта — система геометрических ограничений: параллельность, перпендикулярность, заданное расстояние, касание, симметрия. Дорожка, которая должна быть параллельна краю платы, остаётся параллельной при любых изменениях контура. Произвольные углы — следствие этой архитектуры, а не исключение из правил.

Прямоугольная сетка при этом остаётся доступной как удобный способ работы в простых случаях.

### Производительность

- Рендеринг через GPU: инстансированная отрисовка одинаковых объектов (пады, via одного типа), VBO для дорожек.
- R-tree для всех spatial queries: подсветка цепи, поиск объектов под курсором, proximity DRC — без полного перебора.
- Инкрементальный пересчёт полигонов: при изменении одного объекта пересчитывается только затронутая область.

### DRC

Работает инкрементально, в реальном времени. Правила выразительные: зазоры могут зависеть от типа цепей, разности потенциалов, слоя. Нарушение видно сразу, не по запросу.

### Прочий функционал платы

- Дифференциальные пары и length matching.
- Полный контроль stackup с расчётом импеданса.
- Blind/buried via для многослойных плат.
- Групповые операции с компонентами: выравнивание, распределение, вращение.

## 3D-редактор

3D — не отдельная программа и не просмотрщик. Это режим работы внутри того же инструмента.

- **Двунаправленное редактирование.** Перемещение компонента в 3D-виде сдвигает его футпринт в редакторе платы. Одна операция, один undo.
- **Локальные библиотеки 3D-моделей.** Никакой зависимости от облачных сервисов.
- **Импорт корпуса.** STEP-файл корпуса устройства импортируется и используется как контейнер: инженер видит влезает ли плата, нет ли механических конфликтов. Создавать корпус с нуля внутри инструмента не требуется — только импорт и примерка.
- **Экспорт.** STEP для передачи механикам, и модель с текстурой для визуализации — локально, без Fusion и других облачных сервисов.

## Мультиплатные проекты

Проект может содержать несколько плат, соединённых через разъёмы.

**Верификация разъёмов** — принципиальная фича, которой нет нигде в данном сегменте. Инструмент автоматически проверяет:
- Совпадение распиновки между разъёмом A на плате 1 и разъёмом B на плате 2.
- Корректность ориентации разъёмов при сборке.
- Согласованность сигналов между платами на уровне netlist.

Это переводит класс ошибок "перепутал распиновку разъёма" из категории "найдёшь при сборке прототипа" в категорию "не скомпилируется".

## Гибкие и жёсткогибкие платы

Подход аналогичен sheet metal в механических CAD.

- Зоны сгиба задаются явно с указанием радиуса.
- Flat pattern для производства генерируется автоматически.
- DRC проверяет допустимость дорожек в зоне сгиба: минимальная ширина, запрет via, ориентация дорожек относительно оси сгиба.
- 3D-модель отображает плату в согнутом состоянии.

## Симуляция

Текущие реализации SPICE в EDA-инструментах симулируют аналоговые цепи в вакууме. Цель данного инструмента — приблизиться к модели Proteus: симуляция схемы совместно с поведением микроконтроллера.

Первая версия: полноценная SPICE-симуляция с удобным интерфейсом. Совместная симуляция схемы и прошивки — как следующий этап развития.

## Библиотечная система

Библиотечная система Eagle взята за образец как наиболее прозрачная в классе: связь символа, футпринта и 3D-модели задаётся явно и однозначно.

Расширения:
- Параметрическая база данных компонентов: атрибуты (номинал, допуск, температурный диапазон, артикул производителя) хранятся структурированно.
- Версионирование библиотек: проект фиксирует версию компонента на момент создания.
- Python API для генерации компонентов программно — для параметрических серий (резисторы одного типоразмера, разъёмы одной серии).

---

## Скриптинг и автоматизация

Python API с доступом к полному графу объектов проекта: схема, плата, компоненты, цепи, атрибуты. Это позволяет:

- Автоматизировать расстановку компонентов по алгоритму.
- Генерировать BOM с обращением к внешним базам данных поставщиков.
- Писать кастомные DRC-правила на Python.
- Создавать компоненты программно.

Формат файлов проекта (XML/JSON) дополнительно позволяет обрабатывать проект внешними скриптами без использования API — для тех кто предпочитает работать напрямую с данными.

---

## Формат файлов

- Текстовый (XML или JSON), самодокументированный. Структура читается без документации.
- Git-friendly: диффы между версиями осмысленны и читаемы.
- Пригоден для парсинга внешними скриптами на любом языке.
- Импорт из Eagle (.sch / .brd) и KiCad как приоритетные форматы миграции.

## Платформы

Windows, Linux. macOS — по возможности.

Локальная установка, без обязательного подключения к интернету и без привязки к серверам вендора.

---

## Чего в инструменте нет (сознательно)

- Автороутера. Автороутеры дают плохие результаты, профессионалы ими не пользуются.
- Облачного хранения и совместного редактирования в первой версии.
- Интеграции с PLM-системами (Teamcenter, Windchill) — это enterprise-задача другого масштаба.
